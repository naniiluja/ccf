# Hooks (`plugins/ccf/hooks/*.mjs`)

## Absolute invariants
- **No-build, no-dependency, Windows-clean.** Hooks are `.mjs` ESM run directly with `node` (Node ≥ 18). Do NOT add an npm dependency, do NOT add a transpile/bundle step. All I/O via `node:fs`, `node:path`, `node:child_process` built-ins.
- Type-check with `tsc` using `checkJs` + JSDoc (see `tsconfig.json`). Do NOT switch to `.ts`.
- Paths in `hooks.json` use `node "${CLAUDE_PLUGIN_ROOT}/hooks/<file>.mjs"` — this variable ONLY expands in a hook command.

## I/O contract (packaged in `hooks/lib/io.mjs` — reuse it, don't rewrite)
- **Read stdin**: `readStdinJson()` — always returns `{}` on empty/TTY/parse error so the hook NEVER crashes.
- **Inject context (non-blocking)**: `emitContext(eventName, text)` — prints `{ hookSpecificOutput: { hookEventName, additionalContext } }` then `exit 0`. ONLY for events whose schema accepts `additionalContext`: SessionStart, UserPromptSubmit, PostToolUse, **SubagentStart** (the latter adds the string to the spawned subagent's context before its first prompt; cannot block the spawn). **NOT PreToolUse** — CCF uses only its `permissionDecision`/`permissionDecisionReason` (grounded, PreToolUse ALSO accepts `updatedInput`/`additionalContext`, but CCF does not use them — so don't read this as "deny-only") — and **NOT Stop** (no `hookSpecificOutput`/`additionalContext` at all — validator rejects it as "Invalid input").
- **Warn on a prompt (non-blocking, combined channel)**: `emitPromptWarning(context, message)` — prints the COMBINED `{ hookSpecificOutput: { hookEventName: "UserPromptSubmit", additionalContext }, systemMessage }` payload then `exit 0`. Used by `context-guard.mjs`'s WARN mode so the same advice reaches BOTH the model (`additionalContext`) and the user (`systemMessage`) in one emit; UserPromptSubmit only.
- **Advise at Stop (non-blocking)**: `emitSystemMessage(text)` — prints `{ systemMessage }` then `exit 0`. The only non-blocking channel for a `Stop` hook (omitting `decision` lets the stop proceed; a `decision: "block"` would force Claude to keep working).
- **Block a prompt**: `blockUserPrompt(reason)` — writes `reason` to stderr then `exit 2`. UserPromptSubmit only.
- **Deny a tool (PreToolUse)**: `denyTool(reason)` — prints `{ hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "deny", permissionDecisionReason } }` then `exit 0`. PreToolUse blocks via this JSON, NOT exit 2; to ALLOW, simply `exit 0` with no output.

## Exit code semantics (per Claude Code docs)
- `exit 0` — no decision, the flow continues normally (even after emitting additionalContext).
- `exit 2` — **BLOCK**; stderr is sent back as feedback. This is the only blocking mechanism CCF uses.
- `exit 1` — non-blocking error, does NOT block. CCF avoids it (it adds noise without blocking).

## Events CCF currently uses (`hooks.json`)
- `PreToolUse` (matcher `ExitPlanMode`) → `plan-review-gate.mjs`: enforces that a `/ccf-plan` session reviews its plan before presenting it. Reads `transcript_path` (.jsonl) **best-effort** via `lib/review-trace.mjs` (`hasCcfPlanCommand`/`hasSpecCheckerReview`, pure + unit-tested). If it IS a `/ccf-plan` session (a user line contains `/ccf:ccf-plan` or `/ccf-plan`) AND no spec-checker review is found yet → `denyTool(...)`. **The review detector keys on `subagent_type` (contains `ccf-spec-checker`), accepting a spawn tool named EITHER `Task` OR `Agent` (case-insensitive, `SPAWN_TOOL_NAMES` set).** The spawn-tool name is HARNESS-DEPENDENT — documented as `Task` but surfaced as `Agent` in some runtimes; keying the gate on the name alone DEADLOCKED `Agent`-only harnesses (it never saw the review → blocked `ExitPlanMode` forever). `subagent_type` is the stable gate; the name check only confirms it's a spawn block, so a non-review session still has no match → still allowed (no false positive). Any read/parse failure, or a non-CCF session → `exit 0` (ALLOW): the undocumented transcript shape means we NEVER block wrongly, so this is strong enforcement, not 100% hard — `ccf-plan.md` step 6 is the prompt backup. The review the gate waits for now also returns a **premortem lens** (`ccf-spec-checker` projects prospective failures anchored to `PLAN.md` "Closed" + project memory); the gate MECHANISM is unchanged — it still keys on `ccf-spec-checker` and is satisfied by the FIRST such review it finds (a loop-until-clean plan may produce several; the gate requires ≥1, it does not count them).
- `UserPromptSubmit` → `plan-mode-guard.mjs`: only intervenes on prompts containing `/ccf-plan` (namespaced + bare regex); blocks if `permission_mode !== "plan"`. Every other prompt `exit 0` immediately.
- `SessionStart` (matcher `startup|clear|compact`) → `session-start.mjs`: injects the context-first reminder; if CCF-managed, adds a freshness signal + re-loads the active task (in-progress OR in-review, via `findActiveTask`) after `compact`/`clear`.
  - **Freshness signal** (`lib/freshness.mjs`, shared with `updatespec-nudge.mjs`): the PRIMARY signal is git **committer time** (`git log -1 --format=%ct -- <pathspec>` via `spawnSync`, `shell:false` + `encoding:"utf8"`) of the last commit touching code vs spec — NOT mtime, because ordinary git ops (`checkout`/`pull`/`clone`/even a `commit` that touches a code file last) rewrite mtime to "now" regardless of content, which caused a false "spec older than code" nudge. The pure `decideFreshness` only nudges when BOTH git times resolve and `codeGit > specGit`; a `null` git time on either side (not a git repo, git missing, or a path with no commits) **falls back to the depth-limited mtime walk** (`newestMtime`), so freshly-`/ccf-init`-ed non-git projects still work. Best-effort, advisory only, never throws (`gitCommitTime` returns `null` on any error/empty stdout). Decision logic is pure + unit-tested (`lib/freshness.test.mjs`).
- `Stop` → `updatespec-nudge.mjs`: PURELY ADVISORY, never blocks. Surfaces nudges via `emitSystemMessage` (`{ systemMessage }`), NOT `additionalContext` (which `Stop` does not support). Must check `input.stop_hook_active` to avoid loops. Composes **three INDEPENDENT clauses** into the single non-blocking Stop channel (none gates another; all off → emit nothing): **(A) verify-work** — if this SESSION edited a code file but ran no test command (SESSION evidence read best-effort from `transcript_path` via `lib/verify-trace.mjs`: pure `needsVerifyNudge` + the defensive `readTranscriptSignals`/`editedAnyCodeFile`/`isTestCommand` helpers, all unit-tested, never throw), nudge to run the tests/type-check before stopping; **(B) updatespec** — if `specsOlderThanCode` (spec-vs-code staleness across git history, `lib/freshness.mjs`), nudge `/ccf:ccf-check` then `/ccf:ccf-updatespec`; **(C) plan-status-sync** — if this SESSION ran `git commit` (`lib/git-trace.mjs` `committedThisSession`, the SAME role-gated assistant-`tool_use` shell scan as verify-trace, matching `/\bgit\s+commit\b/`) AND `lib/plan.mjs` `findNonDoneTasks` finds PLAN.md rows still not `done` (todo/in-progress/in-review/blocked), nudge to mark each `done` only after its `/ccf-check` + `/code-review`, or fix its status. The verify-work + plan-status-sync signals key on the HARNESS-DEPENDENT shell-tool name (`bash`/`shell`/`powershell`/`pwsh`) — the `SHELL_TOOL_NAMES` set is COPIED into `git-trace.mjs` (it is module-private in `verify-trace.mjs`, not imported cross-file) — and the harness-dependent edit-tool shape, kept broad so a Windows/PowerShell session or a `MultiEdit` is not a missed signal; both transcript scans are ROLE-GATED to `assistant` lines so user prose ("i will git commit") never false-positives. Despite the filename, the hook is the general CCF Stop nudge, not only the updatespec one.
- `UserPromptSubmit` → `context-guard.mjs` (a SECOND object in the existing `UserPromptSubmit` array, alongside `plan-mode-guard.mjs`): warns/blocks on a proactive `/compact` when context enters the degrade zone. Reads `transcript_path` (.jsonl) to compute current context tokens (`message.usage` of the last assistant line: input + cache_creation + cache_read, via `lib/context-usage.mjs` `readContextUsage`); `shouldNudgeCompact` is true when tokens are ≥ ~40% of the model window — capped at an absolute ~300k tokens via `NUDGE_ABS_CAP` (40% of a 1M-native Opus/Sonnet-4.x window = 400k is unreachable before auto-compact, so the cap keeps the nudge useful), i.e. the "dumb zone". Two modes, chosen by `decideGuardAction({ aboveThreshold, hardBlock, isEscape })`:
  - **WARN (default)** — fires every over-threshold prompt, NON-BLOCKING: `emitPromptWarning(modelContext, userMessage)` surfaces the proactive-`/compact` advice through BOTH `systemMessage` (so the USER sees it) and `additionalContext` (so the MODEL is told to relay it) — deterministic, not dependent on the model choosing to repeat it. Hint pre-filled with the active task (in-progress OR in-review, via `lib/plan.mjs` `findActiveTask`).
  - **HARD-BLOCK (opt-in)** — enabled by adding a `--hard-block` argument to the `context-guard.mjs` command in `hooks.json` (the hooks.json entry IS the toggle, like the template test-gate). Over threshold → `blockUserPrompt(reason)` → `exit 2`, the ONLY deterministic block channel for a UserPromptSubmit (a prompt cannot be merely "suggested" away). **Escape hatch**: an `isEscape` prompt — one whose trimmed text starts with `/compact` (the user is already compacting) OR contains the `ccf:override` token — downgrades block→warn so the user is never trapped.
  - **Compact boundary (just after `/compact`):** `readContextUsage` walks the transcript BACKWARDS and returns null when it hits a `compact_boundary` record BEFORE any assistant line carrying `message.usage` — i.e. right after `/compact`, when no new assistant turn exists yet. Otherwise the hook would read the stale PRE-compact assistant usage (large) and wrongly warn on tokens that no longer exist; returning null keeps the hook silent until the first post-compact assistant turn lands, then it measures the (small, post-compact) context normally. The boundary check runs BEFORE the assistant branch so the two cannot be reordered, and uses a NARROW predicate `isCompactBoundary` (`type:"system"` + `subtype:"compact_boundary"` only — it does NOT read `pre_tokens` nor fall back to `compact_metadata`, so a stray field can never silently suppress the warning; trigger manual|auto are both boundaries).
  - **KISS tradeoff (accepted):** firing on `UserPromptSubmit` (not the old `PostToolUse`) loses MID-TURN detection — a single long turn can cross the threshold without a new prompt — but per-prompt firing is naturally rate-limited (≤ once per user turn), so the old temp-file/`session_id` anti-spam dedup (and its pure decision helper) was REMOVED — no OS-temp state remains. The transcript `.jsonl` shape is **undocumented/internal** → read it **best-effort only**: any error → `readContextUsage` returns null and the hook `exit 0`s silently (never breaks a session). `decideGuardAction`/`shouldNudgeCompact`/`buildCompactHint`/`isCompactBoundary` are pure + unit-tested (`lib/context-usage.test.mjs`).
- `SubagentStart` (NO matcher — an internal `shouldInject(input.agent_type)` allowlist filters instead) → `agent-rules-inject.mjs`: injects, into every spawned file-WRITING subagent, a directive to read & obey the project coding rules (`.claude/rules/*` + CLAUDE.md) plus the active output style's CODING rules (persona/tone/narration/emoji EXCLUDED), then self-check before finishing. Output styles modify the MAIN loop's system prompt only and are NOT inherited by subagents (grounded `code.claude.com/docs/en/output-styles` + `/en/sub-agents`), so a spawned implementer would otherwise lose both. The allowlist is the `WRITER_AGENTS` set (`{ccf-implementer}`) — read-only research/review agents get a no-op (`exit 0`). The directive reaches the subagent via `emitContext("SubagentStart", …)` (`additionalContext`, "added to the subagent's context before its first prompt"; it cannot block the spawn). All decisions live in the pure `lib/output-style.mjs` (`shouldInject` / `resolveActiveOutputStyle` / `buildInjectDirective`, unit-tested); the hook is I/O only. Best-effort: every fs read is defensive and ANY error → `exit 0` (we MUST never block a spawn). Prompt backup: `ccf-implementer.md`'s body restates the same rules.
- `SubagentStart` (matcher `"Explore"` — the SECOND object in the `SubagentStart` array, alongside `agent-rules-inject`) → `explore-guide-inject.mjs`: injects a short, **language-agnostic, LSP-conditional** exploration directive into the built-in `Explore` subagent so it prefers SEMANTIC navigation (the `LSP` tool — `workspaceSymbol`/`goToDefinition`/`findReferences`/`documentSymbol`, with "no server → just fall back, don't retry") + ripgrep-backed `Grep` + `Glob` over reading whole files. CCF does NOT own the `Explore` agent's prompt (it is a harness artifact), so injecting context at spawn via `additionalContext` is the only deterministic lever (same channel as `agent-rules-inject`). Unlike `agent-rules-inject`, this hook has NO internal `agent_type` filter — the hooks.json `"Explore"` matcher IS the gate; an accidental over-match only adds harmless LSP guidance. Content lives in the pure `lib/explore-guide.mjs` (`buildExploreGuidance`, unit-tested); the hook is I/O only. Best-effort: ANY error → `exit 0` (we MUST never block a spawn).
  - **The `SubagentStart` array now holds TWO hooks**: `agent-rules-inject` (NO matcher → fires for every spawn, narrowed by its internal `WRITER_AGENTS` allowlist) and `explore-guide-inject` (matcher `"Explore"` → the harness pre-filters by agent type, so no internal filter is needed).

## Hook-writing conventions
- Each hook starts with `#!/usr/bin/env node` + a comment describing the event, mechanism, role.
- Defensive: wrap every `readdirSync`/`statSync`/`readFileSync` in try/catch, on error → skip the entry (the freshness heuristic is only for nudging, must not break the session).
- Limit recursion depth when walking the tree (e.g. `newestMtime(dir, depth)`); always skip `node_modules` and `.git`.
- Hooks run synchronously and block Claude — keep them fast; `hooks.json` sets a `timeout` (seconds) per hook.
- When adding a new hook: add an entry to `hooks.json` + a comment, reuse `io.mjs`, add the file to `tsconfig.json`'s `include` if it doesn't match the glob.
- **Do NOT declare `"hooks": "./hooks/hooks.json"` in `plugin.json`.** Current Claude Code (v2.1.x) auto-loads `hooks/hooks.json` from the standard location; the `manifest.hooks` field is only for *additional* hook files at a non-standard path (e.g. `./config/hooks.json`). Pointing it back at the standard `hooks/hooks.json` loads the file twice → fatal `Duplicate hooks file detected` and the plugin fails to load. Verify with `claude plugin validate plugins/ccf`. (History: a much older Claude Code did NOT auto-discover and required the field — that behavior was reversed; do not re-add it.)

## Template-shipped hooks (`templates/root/.claude/hooks/*` — a DIFFERENT artifact)
A hook TEMPLATE (e.g. `test-gate.mjs.tmpl`, the opt-in Stop test-gate) is instantiated by `/ccf-init` INTO a target project — it is NOT a CCF-repo hook and obeys different rules:
- **It MAY `exit 2` to BLOCK** (the repo's own Stop hook is deliberately advisory; a target-project opt-in gate is allowed to block — that is its mandate). State this in the file so it isn't "fixed" to advisory.
- **Self-contained**: a target project must not import from the CCF plugin, so COPY the `io.mjs`/`verify-trace.mjs` pattern inline. The ONLY shared piece is a real `.mjs` lib (no `.tmpl`) under `templates/**/lib/` holding the pure decision — so its branchy logic is `node --test`-covered (run `node --test "plugins/ccf/templates/*/.claude/hooks/lib/*.test.mjs"`; the dotted `.claude` MUST be spelled out — a `**` glob skips hidden dirs → 0 tests). Add that lib path to `tsconfig.json` `include`.
- **Hook command uses `$CLAUDE_PROJECT_DIR`** (project root), NOT `${CLAUDE_PLUGIN_ROOT}` (plugin-scoped) — they are different scopes. Presence of the entry in the project's `hooks.json` IS the on/off toggle (no separate settings.json flag).
- **`/ccf-init` must instantiate ALL three** (`test-gate.mjs.tmpl`→`test-gate.mjs`, `hooks.json.tmpl`→`hooks.json`, and the lib **verbatim**), STRIPPING `.tmpl`. Omitting the lib → the hook crashes on a missing import; not stripping `.tmpl` → the loader never finds `hooks.json`.
- **Transcript-scan correctness (two bugs already fixed — do not regress):** (1) **role-gate the scan** — read `tool_use` only from `assistant` lines and `tool_result` only from `user` lines (harness-generated tool output), NEVER scan a user's typed prose, else a chat message containing "FAILED" flips the gate. (2) **a test-FAILED regex must NOT match a passing summary** — "5 passed, **0 failed**" is success; require a failure COUNT `[1-9]\d*` (never `\d+`, which matches the literal 0) and drop the bare word `FAIL`/`FAILED`. Keep this regex in the tested lib (`isFailureText`), not inlined.
